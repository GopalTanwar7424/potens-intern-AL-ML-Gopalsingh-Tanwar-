# Triage Agent with Real Tool Calling

An agentic system that takes free-text support tickets and produces a
structured triage decision using **real Anthropic tool-calling** — the
model itself decides which tools to call, with what arguments, and how
to weigh their output. There is no keyword/if-else shortcut standing
in for the classification.

## Output contract

Every ticket produces:

```json
{
  "category": "Account & Security",
  "priority": "P0",
  "next_tool": "draft_acknowledgment",
  "why": "...",
  "reasoning": ["...ordered natural-language trace..."],
  "confidence": 0.79,
  "tool_calls": [ {"tool_name": "...", "tool_input": {...}, "tool_output": {...}} ],
  "escalated": false,
  "engine": "claude"
}
```

## Taxonomy

**Categories (6):** Billing · Technical Bug · Account & Security ·
Feature Request · Complaint / Escalation · General Inquiry

**Priority:**
- **P0** — urgent / active harm: security breach, data loss, money at
  risk, customer actively threatening to churn.
- **P1** — important, blocking the user, needs same/next-business-day
  attention.
- **P2** — normal queue: questions, minor issues, feature requests.

## The three (+1) tools

| Tool | Purpose |
|---|---|
| `kb_lookup(query)` | Searches a small internal knowledge base for known issues/solutions. Real similarity search (`difflib`), not a lookup the agent can just echo as its final answer. |
| `similar_ticket_search(query)` | Searches a past-ticket archive for similar incidents and how they were actually resolved (category, priority, resolution). |
| `draft_acknowledgment(category, priority, customer_name)` | Drafts a ready-to-send customer reply once a decision is made. |
| `request_human_review(reason, priority)` *(stretch)* | Low-confidence escalation hand-off to a human queue. |

All four are genuine Python functions in `tools.py` with their own
logic and data — see that file for the small KB / past-ticket
datasets. The LLM only sees tool *names and schemas*; it must call
them to get evidence.

## Architecture

```
agent.py            <- real Claude tool-calling loop (messages.create + tools=[...])
tools.py             <- the 4 real callable tools
schemas.py           <- TriageOutput / ToolCallRecord dataclasses, fixed taxonomy
mock_llm.py           <- OFFLINE FALLBACK ONLY (see below) — same tools, heuristic brain
run_examples.py      <- runs all 10 examples, writes examples/outputs/*.json
baseline.py           <- stretch: single-prompt classifier, no tools, for comparison
streamlit_app.py      <- stretch: renders the reasoning trace as a tree
examples/
  inputs.json         <- the 10 test tickets + hand-labeled expected category
  outputs/*.json       <- one file per ticket (input + full trace + decision)
  combined_results.json
  baseline_results.json
```

### Reasoning trace

`agent.py` keeps every assistant text block and every tool call/result
in order, so nothing about the decision is hidden. `TriageOutput.reasoning`
flattens this into a readable ordered list; `full_trace` in the saved
JSON keeps the raw structured version. Nothing in the pipeline emits a
final answer without at least the model's own text reasoning attached.

### Low-confidence escalation (stretch, implemented)

If the model calls `request_human_review`, or if its self-reported
`confidence` is below 0.5, `next_tool` is forced to
`request_human_review` and `escalated: true` is set on the output —
see `EX-07`..`EX-10` in the saved mock-mode examples for what that
looks like end to end.

## Running it for real

```bash
pip install -r requirements.txt
export ANTHROPIC_API_KEY=sk-ant-...
python agent.py "My card was charged twice this month, please fix it"
python run_examples.py          # regenerates examples/outputs/*.json using the real agent
python baseline.py              # stretch: single-prompt baseline, same 10 tickets
streamlit run streamlit_app.py  # stretch: visual reasoning tree
```

## ⚠️ Mock mode — why it's here and how to tell it apart

This project was assembled in a sandbox with **no network access**, so
the `/examples` outputs currently in this repo were generated by
`mock_llm.py`, not by Claude. `mock_llm.py`:

- calls the exact same real tool functions from `tools.py`,
- produces the exact same output schema and trace shape,
- but its "brain" (the thing deciding category/priority from the
  tools' evidence) is a transparent scoring heuristic, not an LLM.

Every mock output is tagged `"engine": "mock"` so it's never confused
with a real run (real runs are tagged `"engine": "claude"`). **Set
`ANTHROPIC_API_KEY` and re-run `python run_examples.py`** to regenerate
`/examples` with genuine LLM-driven tool calling — `agent.py` will
automatically use the real Claude path the moment a key is present, no
code changes needed. The mock engine exists purely so the project is
inspectable/runnable without a key; it is not the graded system.

## Baseline comparison (stretch)

`baseline.py` runs the same 10 tickets through a single-prompt
classifier with **no tools and no evidence gathering**, and reports
category accuracy against the same hand-labeled expectations in
`examples/inputs.json`, side by side with the tool-calling agent's
accuracy from `run_examples.py`. The interesting failure mode to watch
for: the baseline has no way to *cite* why it chose a category (no KB
article, no past ticket) and no escalation path for genuinely
ambiguous tickets — it always guesses.

## Reasoning-tree UI (stretch)

`streamlit_app.py` has two modes: browse the saved `/examples` runs as
an expandable tree (ticket → reasoning/tool-call steps in order →
final decision), or type in a brand new ticket and watch the agent run
live.
